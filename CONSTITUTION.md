# The BuzzPi Constitution

## Preamble

The BuzzPi Platform exists to make managing devices effortless. This constitution defines the principles that guide every decision in the project. These articles rarely change. Governance evolves; constitutions endure.

---

## Article I — Users First

All architectural decisions must reduce friction for users. If a technical choice makes the system easier to build but harder to use, it is the wrong choice. The user's experience is never traded for developer convenience.

---

## Article II — Open by Default

Specifications are open. Reference implementations are open. Documentation is open. The BuzzPi Protocol (BPP) is an open specification, not a proprietary interface. Anyone may implement it without asking permission. The BuzzPi Agent is one reference implementation, not the only implementation.

---

## Article III — Protocol Stability

Breaking changes to BPP require an RFC, a documented migration strategy, protocol version negotiation, and a deprecation window. Clients and agents at different versions must be able to communicate within a supported range. Stability is the foundation of trust for third-party implementers.

---

## Article IV — No Vendor Lock-In

The cloud is optional. The protocol works locally. Users own their data and their devices. No feature requires a BuzzPi-operated service to function. Cloud services enhance the experience but are never required for core operation.

---

## Article V — Clients Are Equal

BuzzPi Android is not privileged over any other client. The desktop client, CLI, web dashboard, and any third-party client implement the same protocol with the same capabilities. The protocol is the contract, not any particular implementation.

---

## Amendment

Amendments to this constitution require approval by a two-thirds majority of Core Maintainers following the RFC process. Amendments are rare by design.
