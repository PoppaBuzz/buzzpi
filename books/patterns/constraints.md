# Patterns Design Constraints

These constraints define how BuzzPi Patterns are documented. They ensure patterns remain useful, consistent, and discoverable.

- Every pattern follows the same template: Problem, Solution, Tradeoffs, Examples, Related Patterns.
- A pattern describes a recurring problem and BuzzPi's solution. It is not implementation instructions.
- Patterns are cross-referenced. No pattern is an island.
- Patterns are written for contributors, not end users. They teach how BuzzPi thinks.
- A pattern is only added after it has been applied in at least two places. Premature patterns are speculation.
- Patterns are living documents. When the solution evolves, the pattern is updated.
- Tradeoffs are mandatory. A pattern that claims no tradeoffs is incomplete.
