# BPP Chapter 6: WebRTC Transport

**Layer:** Transport  
**Status:** Draft  
**Version:** 1.0.0

WebRTC is the primary transport for data-plane communication between client and device. It provides peer-to-peer connectivity, end-to-end encryption, and support for both data channels and media streams.

## Overview

WebRTC is used for all direct client-to-device communication:

- **Data channels:** terminal I/O, file transfer, service messages, events
- **Media tracks:** screen streaming (H.264 video)

WebRTC provides:
- **End-to-end encryption** via DTLS-SRTP (mandatory, always on)
- **NAT traversal** via ICE/STUN/TURN
- **Peer-to-peer** connectivity when possible (lowest latency)
- **Relay fallback** via TURN when P2P is impossible

## ICE Configuration

### STUN Servers

Clients and devices use standard public STUN servers for NAT discovery:

```json
{
  "ice_servers": [
    {
      "urls": [
        "stun:stun.l.google.com:19302",
        "stun:stun1.l.google.com:19302"
      ]
    }
  ]
}
```

STUN is used for address discovery only — it does not route traffic.

### TURN Server

When P2P connectivity fails, traffic routes through the BuzzPi TURN server:

```json
{
  "ice_servers": [
    {
      "urls": [
        "turn:jphat.net/buzzpi/turn:3478",
        "turn:jphat.net/buzzpi/turn:3478?transport=tcp"
      ],
      "username": "<session_id>",
      "credential": "<temporary_turn_credential>"
    }
  ]
}
```

TURN credentials:
- Generated by the Relay Server per-session
- Time-limited (valid for the duration of the session)
- Tied to a specific device ID and client ID
- Single-use (new credentials for each connection attempt)

## Signaling

WebRTC signaling (SDP exchange, ICE candidate exchange) happens over the WebSocket connection to the Relay Server:

### SDP Exchange

```
Client                              Relay Server                            Device
  │                                      │                                   │
  │  relay.sdp.offer                     │                                   │
  │ ────────────────────────────────────►│                                   │
  │                                      │  relay.sdp.offer                  │
  │                                      │ ─────────────────────────────────►│
  │                                      │                                   │
  │                                      │  relay.sdp.answer                 │
  │                                      │ ◀─────────────────────────────────│
  │  relay.sdp.answer                    │                                   │
  │ ◀────────────────────────────────────│                                   │
  │                                      │                                   │
```

### ICE Candidate Exchange

```
  │                                      │                                   │
  │  relay.ice.candidate                 │                                   │
  │ ────────────────────────────────────►│                                   │
  │                                      │  relay.ice.candidate              │
  │                                      │ ─────────────────────────────────►│
  │                                      │                                   │
  │                                      │  relay.ice.candidate              │
  │                                      │ ◀─────────────────────────────────│
  │  relay.ice.candidate                 │                                   │
  │ ◀────────────────────────────────────│                                   │
  │                                      │                                   │
```

After ICE candidate exchange completes, the WebRTC connection is established directly between client and device (or through the TURN server).

## Data Channels

### Configuration

All data channels use the same WebRTC data channel configuration:

```json
{
  "ordered": true,
  "maxRetransmits": 0,
  "protocol": "bpp/data-v1"
}
```

- **Ordered:** true — messages are delivered in order
- **Unreliable:** yes (maxRetransmits: 0) — each message is sent once; if the channel drops, the connection is re-established
- **Protocol:** "bpp/data-v1" — identifies this as BPP data

### Channel Labels

Multiple data channels are established per connection, one per service:

| Label | Direction | Purpose |
|-------|-----------|---------|
| `terminal` | Bidirectional | STDIN/STDOUT for terminal sessions |
| `services` | Bidirectional | Service management messages |
| `files` | Bidirectional | File transfer (chunked) |
| `events` | Device → Client | Real-time event stream |
| `input` | Client → Device | Remote keyboard/mouse input |
| `extension.&lt;name&gt;` | Bidirectional | Extension-specific data |

All channels are established during connection setup. The client and device negotiate which channels to open based on capability discovery.

### Message Format on Data Channels

Data channel messages use a compact binary format (not JSON):

```
┌─────────────────────────────────────────────┐
│ Message on Data Channel                      │
├──────────┬──────────────────────────────────┤
│ Length   │ Payload                          │
│ (4 bytes)│ (variable, up to 64KB)           │
├──────────┴──────────────────────────────────┤
│ Length: Big-endian uint32                    │
│ Payload: Protobuf-encoded BPP message       │
└─────────────────────────────────────────────┘
```

Protobuf is preferred over JSON on data channels for:
- Smaller message size (~40% reduction vs JSON)
- Faster serialization/deserialization
- Strong typing (schema-defined messages)

JSON remains available as a fallback (indicated by a flag byte before the length).

### Flow Control

Data channels use WebRTC's built-in flow control (SCTP). The sender MUST respect the receiver's advertised receive window:

- Sender monitors buffered amount
- When `bufferedAmount` exceeds 64KB, sender pauses and waits for drain event
- Receiver processes messages and the SCTP window opens naturally

## Media Track (Screen Streaming)

### Video Configuration

```json
{
  "video": {
    "codec": "H.264",
    "profile": "Constrained Baseline",
    "packetization": "1 (Single NAL Unit)",
    "resolution": {
      "max_width": 1920,
      "max_height": 1080
    },
    "frame_rate": {
      "max": 30,
      "min": 5
    },
    "bitrate": {
      "max": 5000000,
      "min": 500000
    }
  }
}
```

### Adaptive Quality

The sender monitors WebRTC statistics and adjusts encoding parameters:

| Metric | Adjustment |
|--------|------------|
| Packet loss > 5% | Reduce bitrate by 20% |
| Round-trip time > 300ms | Reduce frame rate by 5fps |
| Available bandwidth < 1Mbps | Reduce resolution to 720p |
| Available bandwidth < 500Kbps | Reduce resolution to 480p |

Adjustments are made every 5 seconds, with a 10-second cooldown between changes to prevent oscillation.

## Connection Type Detection

The client and device report the actual connection type after ICE negotiation:

```json
{
  "connection_type": "p2p",
  "local_candidate_type": "host",
  "remote_candidate_type": "srflx",
  "selected_candidate_pair": "host:srflx",
  "round_trip_time_ms": 12
}
```

| Connection Type | Meaning | Typical Latency |
|-----------------|---------|-----------------|
| `p2p` | Direct peer-to-peer (STUN only) | <5ms LAN, <50ms WAN |
| `p2p_relay` | P2P with TURN as backup | <5ms LAN, <50ms WAN |
| `relay` | TURN relay only (no P2P) | +50-100ms vs P2P |

The connection type is not user-visible but is logged for debugging and monitoring.

## Connection Teardown

Either side can tear down the WebRTC connection:

1. Send `relay.device.disconnect` (via WebSocket signaling)
2. Close all data channels
3. Close the media track
4. Close the RTCPeerConnection

On unexpected disconnection:
1. Both sides attempt to detect the loss (ICE restart or heartbeat timeout)
2. Client attempts reconnection with fresh signaling
3. Device waits for incoming connection request
4. If reconnection fails after 3 attempts, connection is marked as failed
