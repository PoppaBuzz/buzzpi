# BPP Chapter 1: Pairing

**Layer:** Identity  
**Status:** Draft  
**Version:** 1.0.0

Pairing establishes trust between a client and a device. It is a one-time operation. Once paired, the client and device can authenticate each other without repeating the pairing process.

## Overview

Pairing follows the **trust-on-first-use (TOFU)** pattern with an out-of-band verification step:

1. Device generates a time-limited pairing code
2. User enters the code on the client (or scans a QR code)
3. The code proves the user has physical or local-network access to the device
4. Upon verification, client and device exchange identity keys
5. The device is now paired with the user's account

Pairing requires:
- The client and device to be connected to the Relay Server (or on the same local network)
- The device to be in pairing mode (unpaired or explicitly set to pair)
- A human to enter or scan the pairing code

## Pairing Code

### Format

The pairing code is 6 alphanumeric characters, case-insensitive, excluding ambiguous characters:

```
Code characters: ABCDEFGHJKLMNPQRSTUVWXYZ 23456789
(Excluded: I, O, 0, 1 — to avoid visual ambiguity)

Example: A3X9K7
```

### Lifetime

- A pairing code is valid for **5 minutes** from generation
- After expiry, the device generates a new code
- A code can be used exactly once (successful or failed attempt consumes it)
- After a failed attempt, the device generates a new code immediately

### Generation

The code is generated by the device's Runtime using a cryptographically secure random number generator. The code is:

1. Generated on device startup (if unpaired)
2. Displayed on stdout and (optionally) on an attached display
3. Printed to the device log
4. Held in memory only — never persisted to disk

### Verification

1. Client sends the code to the Relay Server: `relay.pair.verify`
2. Relay Server forwards the verification request to the device's Runtime
3. Runtime compares the submitted code against the currently active code
4. If matched: pairing proceeds to key exchange
5. If mismatched: pairing fails, device generates a new code
6. If expired: pairing fails, device generates a new code

## Key Exchange

After code verification, the client and device exchange Ed25519 public keys:

### Client Key

The client generates an Ed25519 keypair (or retrieves an existing one from secure storage). The public key is sent to the device via the Relay Server.

```
ClientKey = Ed25519_PublicKey(client_identity_seed)
```

The client_identity_seed is:
- Generated on first app launch
- Stored in the platform's secure key storage (Android Keystore, iOS Keychain, OS keychain)
- Never transmitted to any server
- Used for all devices paired with this client

### Device Key

The device generates an Ed25519 keypair on first Runtime startup. The public key is sent to the client via the Relay Server.

```
DeviceKey = Ed25519_PublicKey(device_identity_seed)
```

The device_identity_seed is:
- Generated on first Runtime startup
- Stored at `/var/lib/buzzpi/identity.key`
- Never transmitted to any server
- Used for all clients paired with this device

### Key Storage

After exchange, both sides store:

**Client stores:**
- Device ID (UUID v7)
- Device's public key (Ed25519)
- Device's friendly name
- Pairing timestamp

**Device stores:**
- Client's public key (Ed25519)
- User's account identifier
- Pairing timestamp

## Security Considerations

| Threat | Mitigation |
|--------|------------|
| Pairing code interception (local network) | Code is short-lived (5 min), single-use |
| Relay Server compromise during pairing | Pairing messages are forwarded but not decrypted; key exchange is end-to-end |
| Offline brute force of pairing code | 6 chars from 30-char alphabet = 729M combinations; rate-limited to 3 attempts per code |
| Man-in-the-middle during pairing | Code provides out-of-band verification; keys are exchanged over TLS-encrypted WebSocket |
| Device impersonation | Device's identity key is permanent; impersonator cannot present the correct key |

## Post-Pairing

After successful pairing:

1. Device appears in the client's device list as ONLINE (or OFFLINE if disconnected)
2. Client can initiate connections to the device at any time
3. Device trusts this client for future authentication (see Chapter 2)
4. Pairing can be revoked at any time (see Chapter 3)
